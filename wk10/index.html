MDN Form Validation

  • Why validate?
    • Get correct data in proper format
    • To enforce security (such as passwords)
    • To protect ourselves from malicious code
  • Form Validation
    • Server Side
    • Client Side
      • HTML5 form controls
        • required - form cannot be submitted if this field is not filled out
        • minlength or maxlength - min and max length of string in this field
        • min and max - min and max number values in this field
        • type - defines what kind of data in this field
        • pattern - gives a pattern in the form of a regular expression that the data needs to match
          • a -- matches one character that is 'a'
          • abc -- matches 'a', followed by 'b', followed by 'c'
          • ab?c -- matches 'a', optionally 'b', followed by 'c' ('ac' or 'abc')
          • ab*c -- matches 'a', optionally 'b'*x, followed by 'c' ('ac', 'abc' or 'abbbc', etc.)
          • a|b -- matches 'a' or 'b'
          • abc|xyz -- matches 'abc' or 'xyz'
      • Valid data matches :valid CSS pseudo-class and allows the form to be submitted.
      • Invalid data matches :invalid CSS pseudo-class and displays an error instead of submitting form.
      • JavaScript validation
        • Constraint Validation API
          • Form elements:
            • HTMLButtonElement
            • HTMLFieldSetElement
            • HTMLInputElement
            • HTMLOutputElement
            • HTMLSelectElement
            • HTMLTextAreaElement
          • Properties:
            • validationMessage - explains what did not validate
            • validity - returns validityState property:
              • patternMismatch - true if not matching pattern
              • tooLong - true if longer than specified maxlength
              • tooShort - true if shorter than minlength
              • rangeOverflow - true if greater than max
              • rangeUnderflow - true if less than min
              • typeMismatch - true if incorrect syntax (not a valid email or url)
              • valid - true if matching requirements
              • valueMissing - true if required value is missing
            • willValidate - true if elemnt will be validated upon submission
          • Methods:
            • checkValidity() - true if no validity issues
            • setCustomValidity(message) - show custom error message

    Using Fetch

    A basic fetch request looks like this:

    fetch('http://example.com/movies.json')
                .then(response => response.json())
                .then(data => console.log(data));

    If only one argurment, it should be the url/path to the resource being requested. The response comes back as HTTP and needs the json() method to extract the JSON data. If a second parameter is used, init gives control over settings such as: credentials, method, headers, cache and mode.

    Fetch can also POST JSON-encode data (using POST or PUT as a method).

    Files can be uploaded from a form using Fetch.

    A successful fetch will have a Response.ok of 'true' and the promise will be resolved.

    You can create your own headers for fetch using a constructor. This is where you can double check the content type.

    Response Objects

    • Response.status - integer containing response status code (default = 200)
    • Response.statusText - string which give HTTP status code (default = "")
    • Response.ok - shorthand check for status within 200-299 range, inclusive. Boolean returned.

    Body data may be included in either requests or responses. It may be an instance of types: ArrayBuffer, ArrayBufferView, Blob/File, string, URLSearchParams or FormData.

    The body can be extracted using the following methods: arrayBuffer(), blob(), json(), text() or formData().

    Support for Fetch can be detected by checking the window or worker scope for Headers, Request, Response or fetch(). A Polyfill is available for browsers that do not support fetch.
